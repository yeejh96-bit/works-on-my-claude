# works-on-my-claude v1.10.0

코딩을 몰라도, Claude Code가 **꾸준히 잘 일하게 만드는 최소 운영체계**를 빈 폴더에 깐다 — `/womc` 한 번으로.

`/womc` 한 번이면 — 불변 작업 규칙(CLAUDE.md) · 프로젝트 명세(SPEC.md) · 조사 위임(explore) · 검증 위임(verify) ·
한 번에 하나씩 구현하는 운영 절차(plan-feature 스킬) · 규칙 생성(make-rule 스킬) 이 한 묶음으로 세팅된다. 스택도 MCP도 강제하지 않는다.

> 골격만 까는 게 아니다. 만드는 동안 **계획(PLAN)·진행상태(TASKS)·검증**을 Claude가 알아서 만들고 갱신한다.
> 단, 이 파일들은 **필요할 때만** 등장해 평소 컨텍스트를 무겁게 하지 않는다.

## 무엇을 푸는가
새 프로젝트를 시작할 때마다 `CLAUDE.md`, 명세 파일, `.claude/` 폴더 구조를 **0에서 손으로 만드는 번거로움**을 없앤다. 빈 폴더에서 `/womc` 만 입력하면 매번 똑같은 골격이 깔린다.
그리고 깐 뒤에도 끝이 아니다 — 기능을 만들기 시작하면 계획·진행상태·검증까지 **스스로 굴러가는 최소 운영체계**가 된다.

## 설치

Claude Code 안에서 두 줄이면 끝이다.

```
/plugin marketplace add yeejh96-bit/works-on-my-claude
/plugin install womc@works-on-my-claude
```

설치 후 `/reload-plugins` 를 실행하면 `/womc` 가 바로 잡힌다.

### 수동 설치 (플러그인을 쓰지 않을 때)
이 레포의 `commands/womc.md` 를 내려받아 아래 위치에 두어도 된다.
- Mac / Linux: `~/.claude/commands/womc.md`
- Windows: `C:\Users\<사용자명>\.claude\commands\womc.md`

특정 프로젝트에서만 쓰려면 `<프로젝트>/.claude/commands/womc.md` 에 둔다.

## 사용법
1. 새(빈) 프로젝트 폴더를 만든다.
2. 그 폴더에서 Claude Code를 실행한다.
3. 프롬프트에 `/womc` 를 입력한다.
4. 현재 폴더에 골격이 생성된다. 이후 `SPEC.md` 를 채우고 → 계획을 짜고 → 한 기능씩 구현한다.

`/womc` 는 **구조 생성까지만** 한다. SPEC를 대신 채우거나 코드를 짜지 않는다.

> ※ `/womc` 실행 중 파일 생성 권한을 물어보면 "허용"을 선택하세요.

## 이미 깐 프로젝트를 새 버전으로 갱신하기
`/womc` 는 이미 있는 파일을 **건너뛰기** 때문에, 플러그인을 업데이트해도 예전에 깐 프로젝트의 골격은 저절로 바뀌지 않는다. 갱신하려면:
1. 플러그인을 새 버전으로 업데이트한다. Claude Code 안에서:
   ```
   /plugin marketplace update works-on-my-claude
   /plugin install womc@works-on-my-claude
   ```
   (마켓플레이스 정보를 새로고침한 뒤 다시 설치하면 최신 버전이 잡힌다. `/reload-plugins` 까지 하면 확실하다.)
2. 그 프로젝트에서 `/womc update` 를 실행한다 — **불변 골격만** 최신으로 교체되고,
   직접 채운 파일(`SPEC.md` · `PLAN.md` · `TASKS.md` · `.claude/rules/` · `.gitignore`)과
   `settings.json` 에 직접 추가한 허용 항목은 그대로 보존된다. 파일을 손으로 지울 필요가 없다.

## 생성되는 구조
```
현재폴더/
├─ CLAUDE.md          # 항상 적용되는 불변 작업 규칙 (끝에서 @SPEC.md import)
├─ SPEC.md            # 프로젝트 명세 빈 템플릿 (헤더 7개만)
├─ HARNESS.md         # 사람이 읽는 구조 안내서 (Claude가 자동으로 읽지 않음 — 토큰 0)
├─ .gitignore         # node_modules / .env / dist 등 (견본 .env.example 은 예외)
└─ .claude/
   ├─ settings.json      # 하네스: 비밀 .env 읽기 차단(deny, 견본 .env.example 은 읽힘) + PowerShell 읽기 전용 명령 허용
   ├─ agents/
   │  ├─ explore.md      # 코드 조사 전용 서브에이전트 — 조사를 별도 컨텍스트로 빼 긴 세션에서 절약
   │  └─ verify.md       # 동작 검증 전용 서브에이전트 — 테스트·실행을 별도 컨텍스트에서, 통과/실패만 보고
   └─ skills/
      ├─ plan-feature/SKILL.md  # 작게 쪼개 하나씩 구현 + 계획/진행상태 관리 + 검증(필요할 때만 로드)
      └─ make-rule/SKILL.md     # "앞으로 ○○하게 해줘" → 적용 범위를 판단해 규칙 파일 생성
```
> `.mcp.json` 은 **만들지 않는다.** MCP는 프로젝트마다 다르므로 필요할 때 직접 추가한다.
> `.claude/rules/` 폴더도 미리 만들지 않는다 — 규칙 파일은 **매 세션 자동 로드**되므로 빈 자리로 두고,
> `make-rule` 스킬이 첫 규칙을 만들 때 생긴다.

## 이게 왜 "운영체계"인가
골격 생성기와 운영체계의 차이는 **깐 뒤에도 일이 굴러가게 하는 장치가 있느냐**다. womc는 각 역할을 나눠 맡긴다.

| 역할 | 맡는 것 | 로드 방식 |
|---|---|---|
| **무엇을** | `SPEC.md` (불변 명세) | 매 세션 항상 (CLAUDE.md가 import) |
| **어떤 순서로** | `PLAN.md` | 필요할 때만 (스킬이 생성·관리) |
| **지금 어디까지** | `TASKS.md` | 필요할 때만 (스킬이 생성·관리) |
| **운영 절차(쪼개기·구현·검증)** | `plan-feature` 스킬 | 필요할 때만 로드 |
| **조사 위임** | `explore` 서브에이전트 | 호출될 때만 별도 컨텍스트 |
| **검증 위임** | `verify` 서브에이전트 | 호출될 때만 별도 컨텍스트 |
| **규칙 만들기** | `make-rule` 스킬 | 필요할 때만 로드 |
| **권한·안전** | `.claude/settings.json` | 항상(.env 읽기 차단 등) |

핵심은 **always-on은 불변 규칙 + 명세(SPEC)만**이라는 절제다. 가변적인 계획·진행상태는 `@import` 하지 않고
**작업하는 동안에만** 스킬이 읽어들였다 내보낸다. 그래서 "운영체계"의 효과(계획·상태 보존)를 얻으면서도
평소 컨텍스트는 가볍게 유지된다.

## 하네스가 토큰·컨텍스트를 아끼는 방식
골격은 "맥락을 잘 채우는 것"에서 그치지 않고, **작업 중 컨텍스트가 불어나는 걸 막는 장치**를 함께 깐다.

- **`explore` 서브에이전트** — 여러 파일을 뒤지는 무거운 조사를 *별도 컨텍스트*에서 끝내고 **결론만** 메인 대화로 가져온다. 조사 자체도 토큰을 쓰므로 총량이 줄어드는 건 아니지만, 메인 컨텍스트가 불어나지 않아 **긴 세션에서 컴팩션 비용과 재조사를 아낀다.**
- **`verify` 서브에이전트** — 테스트·실행 검증을 별도 컨텍스트에서 끝내고 **통과/실패만** 가져온다. 긴 실행 로그가 본 대화에 쌓이지 않는다. explore·verify 둘 다 더 싸고 빠른 모델(haiku)로 돌게 지정돼 있어 **비용도 아낀다.**
- **`plan-feature` 스킬** — 큰 작업을 한꺼번에 벌이지 않고 작은 단계로 쪼개 하나씩 진행하게 해, 한 세션에 컨텍스트가 터지는 걸 막는다. 스킬은 **필요할 때만 로드**돼 평소 컨텍스트를 차지하지 않는다.
- **CLAUDE.md의 "컨텍스트·토큰 아끼기" 규칙** — 위 습관(무거운 조사·검증은 위임, 한 번에 한 기능, 기능이 끝나면 새 세션)을 매 세션 Claude에게 상기시킨다.
- **settings.json** — 비밀키 파일(`.env` 와 그 변형) 읽기를 하위 폴더 것까지 차단해 입문자를 보호하되, 비밀이 없는 견본 파일(`.env.example`)은 읽을 수 있게 **비밀이 들어가는 이름만 골라** 막는다(Claude가 "어떤 설정값이 필요한지" 견본에서 파악할 수 있게). Windows(PowerShell)에서 자주 쓰는 읽기 전용 명령도 미리 허용한다. (Bash의 `git status` 등 읽기 전용 명령은 Claude Code가 기본으로 프롬프트 없이 허용하고, Mac/Linux에선 PowerShell 항목이 조용히 무시된다. 차단은 파일 읽기 도구에만 걸리므로, 진짜 중요한 비밀은 프로젝트 폴더 밖에 두는 게 가장 안전하다.)

## 설계 원리
**변하지 않는 것만 내용을 채우고, 프로젝트마다 달라지는 것은 빈 칸으로 두거나 아예 만들지 않는다.**

- 항상 같은 것 (→ 내용을 채움): 코딩 입문자라는 점, "수정은 직접·필요한 만큼만 보여줌", "한국어로 설명" 같은 불변 작업 규칙.
- 프로젝트마다 다른 것 (→ 빈 틀 또는 생성 안 함): 기술 스택·기능은 `SPEC.md` 빈 템플릿으로만, MCP 설정은 아예 생성하지 않음.

골격에 스택·MCP를 미리 박아두면 다음 프로젝트에서 오히려 걸리적거리기 때문이다.

## 파일 로드 방식
- **CLAUDE.md** 는 Claude Code가 켜질 때 **매 세션 자동으로 로드**된다.
- **SPEC.md** 는 CLAUDE.md 맨 끝의 `@SPEC.md` 줄을 통해 **import 되어 함께 딸려 로드**된다.

즉 불변 규칙(CLAUDE.md)과 프로젝트 명세(SPEC.md)가 매 세션 한 묶음으로 Claude에게 전달된다.

- **PLAN.md · TASKS.md** 는 일부러 **import 하지 않는다.** 기능을 만들 때 `plan-feature` 스킬이
  필요한 동안에만 읽고 갱신한다. 평소 컨텍스트에는 올라오지 않아, 진행상태를 남기면서도 토큰을 아낀다.
