너는 지금 현재 작업 폴더에 "프로젝트 기본 골격"을 생성하는 작업만 한다.
아래 규칙을 반드시 지켜라.

## 절대 규칙
- 아래에 적힌 파일/폴더만, 적힌 내용 그대로 생성한다. (매 프로젝트 항상 동일)
- 프로젝트마다 달라지는 것(기술 스택, 사용하는 MCP 등)은 채우지 않는다. 빈 틀만 둔다.
- `.mcp.json` 은 만들지 않는다.
- SPEC 자동 작성, 코드 구현은 하지 않는다. 구조 생성까지만 한다.
- 같은 이름 파일이 이미 있으면 덮어쓰지 말고, 어떤 파일이 이미 있었는지 마지막 안내에서 알려준다.

## 생성할 파일과 내용

### 1) `CLAUDE.md`
아래 내용을 그대로 쓴다.

```markdown
# 작업 규칙 (모든 프로젝트 공통 · 불변)

이 파일은 매 세션 자동으로 로드된다. 아래 규칙은 프로젝트와 상관없이 항상 적용된다.

## 나에 대해
- 나는 코딩을 모른다. 전문 용어를 최소화하고, 필요한 건 풀어서 설명한다.

## 코드 제공 방식
- 코드는 항상 **파일 전체를 통째로** 보여준다. (일부 줄만 바꾸라는 식으로 주지 않는다)
- 어떤 파일인지 파일 경로를 함께 적는다.
- 파일을 새로 만들거나 바꾸면, 그다음 무엇을 하면 되는지 한 줄로 알려준다.

## 설명 방식
- 모든 설명은 한국어로 한다.

## 작업 진행 방식 (한 번에 다 하지 않는다)
큰 요청이 들어와도 **곧바로 전부 구현하지 않는다.** 다음 순서를 지킨다.

1. **먼저 쪼갠다.** 코드부터 짜지 말고, 요청을 작은 단위(task)로 나눈 목록을 만들어 `TASKS.md` 에 적고 사용자에게 보여준다.
2. **어떻게 진행할지 먼저 묻는다.** 코드를 짜기 전에, 쪼갠 task 목록을 보여주면서 **진행 방식**을 사용자에게 물어본다. 두 가지 중 고르게 한다.
   - **(A) 끝까지 한 번에** — 멈추지 않고 모든 task를 순서대로 끝까지 구현한다.
   - **(B) 하나씩 확인받으며** — task 한 개를 끝낼 때마다 멈추고, 다음으로 넘어가도 되는지 확인을 받은 뒤 진행한다.

   사용자가 고르기 전에는 구현을 시작하지 않는다.
3. **하나만 한다.** 어느 방식이든 한 번에 **딱 한 개의 task**만 구현한다. 미리 여러 task를 앞질러 가지 않는다.
4. **task가 끝나면:**
   - 공통: (1) 무엇을 했는지 한 줄, (2) `TASKS.md` 의 해당 항목 체크.
   - **(A) 방식이면** 곧바로 다음 task로 이어서 진행한다. (단, 큰 결정이 갈리는 지점이 나오면 그때는 멈추고 묻는다.)
   - **(B) 방식이면** 다음 task가 무엇인지 알려주고 **멈춘다.** "다음 걸 진행할까요?" 라고 묻고, 사용자가 좋다고 해야 다음으로 넘어간다.
5. **항상 최신으로.** 진행 상황(`[ ]` / `[~]` / `[x]`)을 `TASKS.md` 에 늘 최신 상태로 유지한다.

## 프로젝트 상세
이 프로젝트의 구체적인 내용(스택, 무엇을 만드는지 등)과 진행 상황은 아래 파일에 있다.

@SPEC.md
@TASKS.md
```

### 2) `SPEC.md`
아래 내용을 그대로 쓴다. (헤더만 두고 내용은 비운다)

```markdown
# 프로젝트 명세 (SPEC)

> 이 파일을 채운 뒤 작업을 시작한다. 비어 있으면 먼저 채운다.

## 1. 기술 스택

## 2. 뭘 푸는가

## 3. 핵심 기능 (3~5개)

## 4. 화면 흐름

## 5. 저장할 데이터

## 6. 안 만들 것
```

### 3) `TASKS.md`
아래 내용을 그대로 쓴다. (틀만 두고 task는 비운다)

```markdown
# 작업 목록 (TASKS)

> 큰 작업은 여기서 작은 단위(task)로 쪼갠다.
> 시작 전에 진행 방식을 정한다: (A) 끝까지 한 번에 / (B) task 하나씩 확인받으며.
>
> 표기: `[ ]` 안 함 · `[~]` 하는 중 · `[x]` 끝남

## 지금 하는 일
(아직 없음)

## 할 일 (순서대로)
- [ ] (아직 비어 있음 — SPEC.md 를 채운 뒤, 여기에 task를 잘게 쪼개 적는다)

## 끝난 일
(아직 없음)
```

### 4) `README.md`
아래 내용을 그대로 쓴다.

```markdown
# 프로젝트 구조 안내

이 폴더는 Claude Code가 일을 잘하도록 **컨텍스트 엔지니어링**과 **하네스 세팅**을
미리 갖춘 구조다. 각 파일이 맡은 역할은 다음과 같다.

## 파일별 역할
- **CLAUDE.md** — 프로젝트와 상관없이 항상 적용되는 불변 작업 규칙.
  Claude Code를 켜면 **매 세션 자동으로 로드**된다. (예: 한국어로 설명, 코드는 전체 파일로 제공)
- **SPEC.md** — 이 프로젝트만의 상세 내용(스택, 만들 것, 기능 등). CLAUDE.md 끝에서
  `@SPEC.md` 로 import 되어 **CLAUDE.md에 딸려 함께 로드**된다. 처음엔 빈 템플릿이며 직접 채운다.
- **TASKS.md** — 큰 작업을 작은 단위(task)로 쪼개 **하나씩** 처리하기 위한 작업 목록.
  CLAUDE.md 끝에서 `@TASKS.md` 로 import 되어 **매 세션 현재 진행 상황이 함께 로드**된다.
  덕분에 세션이 끊기거나 길어져도 "어디까지 했는지"가 보존된다.
- **.claude/agents/** — 이 프로젝트 전용 서브에이전트를 두는 곳. (지금은 `README.md` 만 있음)
- **.claude/skills/** — 이 프로젝트 전용 스킬을 두는 곳. (지금은 `README.md` 만 있음)
- **.claude/rules/** — 세부 규칙 문서를 두는 곳. (지금은 `README.md` 만 있음)
- **.gitignore** — git이 무시할 파일 목록(설치물, 비밀키, 빌드 산출물 등).

## 왜 스택·MCP 같은 건 비워 뒀나
기술 스택, 사용하는 MCP 등은 **프로젝트마다 다르다.** 기본 골격에 미리 박아두면
새 프로젝트에서 오히려 방해가 된다. 그래서 변하지 않는 규칙만 채워 두고,
변하는 것은 **빈 칸(SPEC.md)** 으로 두거나 **아예 만들지 않는다(.mcp.json 등).**
필요한 프로젝트에서 그때 직접 추가한다.

## 다음에 할 일
1. `SPEC.md` 를 채운다.
2. `TASKS.md` 에 무엇을 어떤 순서로 만들지 작은 task로 쪼개 적는다.
3. 시작 전에 진행 방식을 정한다 — (A) 끝까지 한 번에 / (B) task 하나씩 확인받으며.
4. 정한 방식대로 **한 번에 하나의 task씩** 구현한다.
```

### 6) `.claude/` 하위 폴더 3개 (`README.md` 로 추적)
다음 3개 파일을 각각 아래 내용 그대로 만든다. (빈 폴더를 빈 파일로 추적하면 권한 분류기에 걸려 생성이 누락될 수 있으므로, 내용이 있는 `README.md` 로 둔다.)

`.claude/agents/README.md`
```markdown
# agents
이 폴더는 이 프로젝트 전용 서브에이전트를 두는 곳이다.
조사·검토처럼 무거운 작업을 별도 컨텍스트에서 처리하고 싶을 때 추가한다.
지금은 비어 있다.
```

`.claude/skills/README.md`
```markdown
# skills
이 폴더는 이 프로젝트 전용 스킬(반복 절차서)을 두는 곳이다.
같은 작업을 반복하게 될 때 추가한다.
지금은 비어 있다.
```

`.claude/rules/README.md`
```markdown
# rules
이 폴더는 세부 규칙 문서를 두는 곳이다.
특정 파일을 만질 때만 적용할 규칙이 생기면 추가한다.
지금은 비어 있다.
```

## 생성이 끝나면
한국어로 짧게 다음 단계를 안내한다. 예시:

> ✅ 기본 골격을 만들었어요.
> 다음 순서로 진행하면 됩니다.
> 1) `SPEC.md` 를 채우기 — 어떤 걸 만들지 정한다.
> 2) `TASKS.md` 에 task 쪼개기 — 무엇을 어떤 순서로 만들지 작은 단위로 나눈다.
> 3) 진행 방식 정하기 — (A) 끝까지 한 번에 / (B) task 하나씩 확인받으며.
> 4) 정한 방식대로 **한 번에 하나의 task씩** 구현하기.
>
> ※ MCP가 필요한 프로젝트라면 그때 `.mcp.json` 을 직접 추가하세요.

(SPEC를 대신 채우거나 코드를 짜지 말 것. 여기서 멈춘다.)
