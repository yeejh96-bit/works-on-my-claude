---
description: 새 프로젝트 폴더에 항상 동일한 기본 골격(컨텍스트 엔지니어링 + 하네스 세팅)을 생성한다
allowed-tools: Write, Read, Glob
---

너는 지금 현재 작업 폴더에 "프로젝트 기본 골격"을 생성하는 작업만 한다.
아래 규칙을 반드시 지켜라.

## 절대 규칙
- 아래에 적힌 파일/폴더만, 적힌 내용 그대로 생성한다. (매 프로젝트 항상 동일)
- 프로젝트마다 달라지는 것(기술 스택, 사용하는 MCP 등)은 채우지 않는다. 빈 틀만 둔다.
- `.mcp.json` 은 만들지 않는다.
- SPEC 자동 작성, 코드 구현은 하지 않는다. 구조 생성까지만 한다.
- 생성하기 전에 먼저 `Glob` 로 현재 폴더에 같은 이름 파일이 이미 있는지 확인한다.
  이미 있는 파일은 **덮어쓰지 말고 건너뛰며**, 어떤 파일이 이미 있었는지 마지막 안내에서 알려준다.

## 생성할 파일과 내용

### 1) `CLAUDE.md`
아래 내용을 그대로 쓴다.

```markdown
# 작업 규칙 (모든 프로젝트 공통 · 불변)

이 파일은 매 세션 자동으로 로드된다. 아래 규칙은 프로젝트와 상관없이 항상 적용된다.

## 나에 대해
- 나는 코딩을 모른다. 전문 용어를 최소화하고, 필요한 건 풀어서 설명한다.

## 코드 제공 방식
- 코드는 항상 **파일 전체를 통째로** 보여준다. (일부 줄만 바꾸라는 식으로 주지 않는다)
- 어떤 파일인지 파일 경로를 함께 적는다.
- 파일을 새로 만들거나 바꾸면, 그다음 무엇을 하면 되는지 한 줄로 알려준다.

## 설명 방식
- 모든 설명은 한국어로 한다.

## 컨텍스트·토큰 아끼기 (하네스 습관)
- 여러 파일을 뒤져야 하는 무거운 조사는 `explore` 서브에이전트에게 맡긴다. 조사는 별도 컨텍스트에서 끝나고 결론만 돌아오므로, 메인 대화에 파일 내용이 쌓이지 않는다.
- 한 번에 한 기능만 진행한다. 큰 작업을 한꺼번에 벌이지 않는다.
- 세션이 길어지면 `/compact` 로 대화를 정리한 뒤 이어간다.

## 프로젝트 상세
이 프로젝트의 구체적인 내용(스택, 무엇을 만드는지 등)은 아래 파일에 있다.

@SPEC.md
```

### 2) `SPEC.md`
아래 내용을 그대로 쓴다. (헤더만 두고 내용은 비운다)

```markdown
# 프로젝트 명세 (SPEC)

> 이 파일을 채운 뒤 작업을 시작한다. 비어 있으면 먼저 채운다.

## 1. 기술 스택

## 2. 뭘 푸는가

## 3. 핵심 기능 (3~5개)

## 4. 화면 흐름

## 5. 저장할 데이터

## 6. 안 만들 것
```

### 3) `README.md`
아래 내용을 그대로 쓴다.

```markdown
# 프로젝트 구조 안내

이 폴더는 Claude Code가 일을 잘하도록 **컨텍스트 엔지니어링**과 **하네스 세팅**을
미리 갖춘 구조다. 각 파일이 맡은 역할은 다음과 같다.

## 파일별 역할
- **CLAUDE.md** — 프로젝트와 상관없이 항상 적용되는 불변 작업 규칙.
  Claude Code를 켜면 **매 세션 자동으로 로드**된다. (예: 한국어로 설명, 코드는 전체 파일로 제공)
- **SPEC.md** — 이 프로젝트만의 상세 내용(스택, 만들 것, 기능 등). CLAUDE.md 끝에서
  `@SPEC.md` 로 import 되어 **CLAUDE.md에 딸려 함께 로드**된다. 처음엔 빈 템플릿이며 직접 채운다.
- **.claude/settings.json** — 하네스(작업 환경) 기본값. 어디서나 안전한 읽기 전용 명령
  (`git status`, `git diff`, `git log`, 파일 목록)을 미리 허용해, 매번 허용 프롬프트가 뜨는 걸 줄인다.
- **.claude/agents/** — 무거운 조사를 별도 컨텍스트에서 처리하는 서브에이전트. `explore` 가 들어 있어
  여러 파일을 뒤지는 조사를 메인 대화 밖에서 끝내고 결론만 가져온다(토큰 절약).
- **.claude/skills/** — 필요할 때만 로드되는 반복 절차서. `plan-feature` 가 들어 있어
  새 기능을 작게 쪼개 한 번에 하나씩 구현하도록 돕는다.
- **.claude/rules/** — 특정 작업에만 적용할 세부 규칙 자리. 자동 로드되지 않으며 `@import` 로 켠다.
- **.gitignore** — git이 무시할 파일 목록(설치물, 비밀키, 빌드 산출물 등).

## 왜 스택·MCP 같은 건 비워 뒀나
기술 스택, 사용하는 MCP 등은 **프로젝트마다 다르다.** 기본 골격에 미리 박아두면
새 프로젝트에서 오히려 방해가 된다. 그래서 변하지 않는 규칙만 채워 두고,
변하는 것은 **빈 칸(SPEC.md)** 으로 두거나 **아예 만들지 않는다(.mcp.json 등).**
필요한 프로젝트에서 그때 직접 추가한다.

## 다음에 할 일
1. `SPEC.md` 를 채운다.
2. 무엇을 어떤 순서로 만들지 계획(PLAN)을 세운다.
3. 한 번에 하나의 기능씩 구현한다.
```

### 5) `.claude/` 하위 폴더와 하네스 파일들
아래 파일들을 각각 그대로 만든다. (빈 폴더는 권한 분류기에 걸려 생성이 누락될 수 있으므로 항상 내용 있는 파일로 둔다.)

`.claude/agents/README.md`
```markdown
# agents
이 프로젝트 전용 서브에이전트를 두는 곳이다.
무거운 조사를 별도 컨텍스트에서 처리해 메인 대화의 토큰을 아낀다.
- `explore.md` — 코드 조사 전용. "어디 있지 / 어떻게 동작하지"를 물으면 여기로 위임된다.
직접 추가하려면 같은 형식의 `.md` 파일을 만들면 된다.
```

`.claude/agents/explore.md` (실제 동작하는 조사 전용 서브에이전트)
```markdown
---
name: explore
description: 코드베이스에서 "이게 어디 있지 / 어떻게 동작하지"를 조사할 때 사용한다. 여러 파일을 읽어야 하는 무거운 조사를 별도 컨텍스트에서 처리하고 결론만 돌려줘, 메인 대화의 토큰을 아낀다.
tools: Read, Grep, Glob
---
너는 코드 조사 전용 에이전트다. 메인 대화의 컨텍스트를 아끼는 것이 존재 이유다.

규칙:
- 요청받은 것만 조사하고, 끝나면 결론만 짧게 돌려준다.
- 파일 내용을 통째로 붙여넣지 않는다. 대신 `파일경로:줄번호` 로 위치만 가리킨다.
- 코드를 수정하지 않는다. 읽기·검색만 한다.
- 확실하지 않으면 "확인 못 함"이라고 분명히 적는다.

출력 형식:
1) 한 줄 요약
2) 핵심 발견 (불릿마다 `파일경로:줄번호`)
3) 다음에 보면 좋을 곳 (선택)
```

`.claude/skills/README.md`
```markdown
# skills
이 프로젝트 전용 스킬(반복 절차서)을 두는 곳이다. 필요할 때만 로드돼 컨텍스트를 아낀다.
- `plan-feature/SKILL.md` — 새 기능을 작게 쪼개 한 번에 하나씩 구현하는 절차.
직접 추가하려면 `스킬이름/SKILL.md` 형식으로 폴더를 만든다.
```

`.claude/skills/plan-feature/SKILL.md` (실제 동작하는 스킬)
```markdown
---
name: plan-feature
description: 새 기능을 하나 추가할 때 사용한다. 큰 작업을 작은 단계로 쪼개 한 번에 하나씩 구현하게 해서, 한 세션에 너무 많은 걸 벌이다 컨텍스트가 터지는 것을 막는다.
---
# 한 기능 추가하기

사용자가 "○○ 기능 만들자"고 하면 이 절차를 따른다.

## 1. SPEC 확인
`SPEC.md` 에 이 기능이 있는지 본다. 없으면 한 줄로 추가를 제안한다.

## 2. 작게 쪼개기
기능을 3단계 이하로 나눈다. 각 단계는 파일 1~2개 수준이어야 한다.
단계 목록을 보여주고 어디부터 할지 정한다.

## 3. 한 단계만 구현
한 번에 한 단계만. 파일은 전체를 보여준다(CLAUDE.md 규칙).
끝나면 다음에 뭘 하면 되는지 한 줄로 안내한다.

## 4. 확인하고 다음으로
한 단계가 동작하는 걸 확인한 뒤에만 다음 단계로 간다.
세션이 길어지면 `/compact` 로 정리하고 이어간다.
```

`.claude/rules/README.md`
```markdown
# rules
특정 파일이나 작업에만 적용할 세부 규칙 문서를 두는 곳이다.
여기 둔 문서는 자동으로 로드되지 않는다. 적용하려면 `CLAUDE.md` 끝에
`@.claude/rules/파일명.md` 한 줄을 더해 import 한다.
지금은 비어 있다.
```

### 6) `.claude/settings.json` (하네스 — 권한 기본값)
아래 내용을 그대로 쓴다. 어느 프로젝트에서나 안전한 **읽기 전용 명령**만 미리 허용해 둔다.
(입문자가 `git status` 한 번 돌릴 때마다 허용 프롬프트에 시달리지 않도록 하는 게 목적이다.
`push`·`reset` 같이 되돌리기 어려운 명령은 일부러 넣지 않는다 — 그건 매번 확인받는 게 맞다.)

```json
{
  "permissions": {
    "allow": [
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(ls:*)",
      "PowerShell(git status*)",
      "PowerShell(git diff*)",
      "PowerShell(git log*)",
      "PowerShell(Get-ChildItem*)"
    ]
  }
}
```

## 생성이 끝나면
한국어로 짧게 다음 단계를 안내한다. 예시:

> ✅ 기본 골격을 만들었어요.
> 다음 순서로 진행하면 됩니다.
> 1) `SPEC.md` 를 채우기 — 어떤 걸 만들지 정한다.
> 2) PLAN 짜기 — 무엇을 어떤 순서로 만들지 계획한다.
> 3) 한 기능씩 구현하기.
>
> ※ MCP가 필요한 프로젝트라면 그때 `.mcp.json` 을 직접 추가하세요.

(SPEC를 대신 채우거나 코드를 짜지 말 것. 여기서 멈춘다.)
