사용자 인자: $ARGUMENTS

- 인자에 `update` 가 있으면 → 맨 아래 **갱신 모드** 절차만 수행한다. (아래 생성 절차는 건너뛴다)
- 인자가 없으면 → 아래 생성 절차(기본)를 수행한다.

너는 지금 현재 작업 폴더에 "프로젝트 기본 골격"을 생성하는 작업만 한다.
아래 규칙을 반드시 지켜라.

## 절대 규칙
- 아래에 적힌 파일/폴더만, 적힌 내용 그대로 생성한다. (매 프로젝트 항상 동일)
- 프로젝트마다 달라지는 것(기술 스택, 사용하는 MCP 등)은 채우지 않는다. 빈 틀만 둔다.
- `.mcp.json` 은 만들지 않는다.
- 코드 구현은 하지 않는다. 구조 생성까지만 한다. SPEC 은 신규 빈 폴더면 빈 틀만 두고, 기존 코드가 있는 폴더면 `explore` 로 훑어 **초안까지만** 채운다(사용자 검토 전제 — 아래 '기존 프로젝트 온보딩' 참고).
- 생성하기 전에 먼저 `Glob` 로 현재 폴더에 같은 이름 파일이 이미 있는지 확인한다.
  이미 있는 파일은 **덮어쓰지 말고 건너뛰며**, 어떤 파일이 이미 있었는지 마지막 안내에서 알려준다.
- 단, **기존 프로젝트 온보딩**일 때는 이미 있는 `CLAUDE.md`·`.claude/settings.json` 을 그냥 건너뛰지 않고 womc 내용을 **덧붙여 병합한다** — 사용자가 쓴 내용은 한 줄도 지우지 않고 추가만 한다(아래 '기존 프로젝트 온보딩' 참고).

## 생성할 파일과 내용

### 1) `CLAUDE.md`
아래 내용을 그대로 쓴다.

```markdown
# 작업 규칙 (모든 프로젝트 공통 · 불변)

이 파일은 매 세션 자동으로 로드된다. 아래 규칙은 프로젝트와 상관없이 항상 적용된다.

## 나에 대해
- 나는 코딩을 모른다. 전문 용어를 최소화하고, 필요한 건 풀어서 설명한다.

## 코드 제공 방식
- 코드 수정은 너(Claude)가 파일에 직접 한다. 나는 복사·붙여넣기 하지 않는다. (바뀐 부분만 정확히 고친다)
- 보여줄 때는 무엇을 왜 바꿨는지 한국어로 설명하고, 필요한 만큼만 보여준다. (작은 파일이면 전체, 큰 파일이면 바뀐 부분 위주 — 토큰 낭비를 막는다)
- 어떤 파일인지 파일 경로를 함께 적는다.
- 파일을 새로 만들거나 바꾸면, 그다음 무엇을 하면 되는지 한 줄로 알려준다.

## 설명 방식
- 모든 설명은 한국어로 한다.

## 적극 위임 (오케스트레이터 방식)
- 너(메인)는 지휘자다. 무거운 일은 직접 하지 말고 전문 서브에이전트에 맡긴 뒤 결론만 받아 조립한다. 메인 대화는 얇게 유지한다. 위임이 기본값이고, 직접 처리는 사소할 때만.
- 조사는 `explore`(여러 갈래면 한 메시지에서 여러 개 병렬), 설계는 `plan`, 구현은 `implement`, 동작 검증은 `verify`, 코드 검토는 `review` 에 맡긴다. 서브에이전트는 결론(`파일경로:줄번호`)만 돌려주므로 파일 내용·로그가 메인에 쌓이지 않는다.
- 위임은 정해진 단계에서만이 아니라 필요하다고 판단하는 그 순간 한다. 개수 상한은 없다 — 독립적인 갈래가 있으면 같은 종류를 여러 개 병렬로 띄운다. 이 5종에 딱 안 맞는 일은 내장 `general-purpose` 에이전트에 **그 자리에서 역할·지시를 담은 프롬프트로** 즉석 위임한다 — 파일을 새로 만들 필요 없이 이 세션에서 필요한 만큼 자유롭게 띄운다. (어떤 역할을 매 세션 되풀이해 쓰고 싶을 때만 `.claude/agents/` 에 파일로 굳히면 되고, 그건 세션을 다시 켜야 인식된다.)
- 코드 수정은 `implement` 서브에이전트가 파일에 직접 하고, 메인은 그 결과를 받아 검토·조립한다(사용자에게 복붙 시키지 않는다는 원칙은 그대로다). 아주 작은 수정은 메인이 직접 해도 된다.
- 한 번에 한 기능만 진행한다. 큰 작업을 한꺼번에 벌이지 않는다.
- 한 기능이 끝나면 새 세션으로 시작하길 사용자에게 권한다(진행상태는 PLAN.md/TASKS.md 에 남는다).
  작업 중간에 대화가 무거워지면 사용자에게 `/compact` 를 제안한다. (`/compact` 는 사용자 명령이라 네가 직접 실행할 수 없다.)

## 프로젝트 상세
이 프로젝트의 구체적인 내용(스택, 무엇을 만드는지 등)은 아래 파일에 있다.

@SPEC.md
```

### 2) `SPEC.md`
아래 내용을 그대로 쓴다. 신규 빈 폴더면 헤더만 두고 내용은 비운다.
(기존 코드가 있는 폴더면, 아래 '기존 프로젝트 온보딩'에서 `explore` 로 훑은 결과로 각 섹션을 채운다 — 초안이며 사용자가 검토·수정한다.)

### 3) `HARNESS.md`
아래 내용을 그대로 쓴다.
(사람이 읽는 안내서다 — Claude가 자동으로 읽지 않으므로 평소 토큰을 쓰지 않는다.
프로젝트 소개용 `README.md` 는 만들지 않는다 — 프로젝트마다 내용이 다르므로, 사용자가 필요할 때 직접 만들거나 Claude에게 부탁한다.)

```markdown
# 하네스 안내 (HARNESS)

이 폴더는 Claude Code가 일을 잘하도록 **컨텍스트 엔지니어링**과 **하네스 세팅**을
미리 갖춘 구조다. 이 파일은 **사람(사용자)이 읽는 안내서**이며, Claude가 자동으로 읽지 않아
토큰을 쓰지 않는다. 각 파일이 맡은 역할은 다음과 같다.

## 파일별 역할
- **CLAUDE.md** — 프로젝트와 상관없이 항상 적용되는 불변 작업 규칙.
  Claude Code를 켜면 **매 세션 자동으로 로드**된다. (예: 한국어로 설명, 수정은 직접 하고 필요한 만큼만 보여줌)
- **SPEC.md** — 이 프로젝트만의 상세 내용(스택, 만들 것, 기능 등). CLAUDE.md 끝에서
  `@SPEC.md` 로 import 되어 **CLAUDE.md에 딸려 함께 로드**된다.
  신규 프로젝트면 빈 템플릿을 직접 채우고, 기존 프로젝트면 `/womc` 가 코드를 훑어 채운 초안을 검토·수정한다.
  (기존 프로젝트면 이미 있던 `CLAUDE.md`·설정에 womc 규칙을 덧붙여 병합하고, 코드 관례를 `.claude/rules/` 로 캡처하며, 구조 정리는 `plan-feature` 로 넘기도록 제안한다.)
- **.claude/settings.json** — 하네스(작업 환경) 기본값. 비밀키 파일(`.env` 와 그 변형)을 하위 폴더 것까지
  Claude가 읽지 못하게 막고(`deny`), 비밀이 없는 견본 파일(`.env.example`)은 읽을 수 있게 둔다.
  Windows(PowerShell)에서 자주 쓰는 읽기 전용 명령도 미리 허용해 둔다.
  (`git status` 같은 Bash 읽기 전용 명령은 Claude Code가 기본으로 프롬프트 없이 허용하므로 따로 적지 않는다.
  Mac/Linux 에서는 PowerShell 항목이 조용히 무시된다. 차단은 파일 읽기 도구에만 걸리므로, 진짜 비밀은 프로젝트 폴더 밖에 두는 게 가장 안전하다.)
- **.claude/agents/** — 무거운 일을 별도 컨텍스트에서 처리하는 전문 서브에이전트 5종. 메인(지휘자)이 조사·설계·구현·검증·검토를
  이들에게 맡기고 결론만 받아 조립하므로, 파일 내용·로그가 메인 대화에 쌓이지 않는다. 독립적인 일은 여러 개를 병렬로 돌린다.
  `explore`(조사·빠른 haiku) · `plan`(설계·품질 위주 opus) · `implement`(구현·유일하게 파일을 직접 고침) ·
  `verify`(동작 검증) · `review`(코드 검토). 읽기 전용 4종에는 수정 도구를 아예 주지 않아 실수로도 파일을 못 고친다.
  각 파일에는 그 에이전트만의 작업 규칙(받아야 할 정보·한국어 보고·결론만 반환 등)이 들어 있다 — 서브에이전트는 CLAUDE.md 를 물려받지 않기 때문이다.
  settings.json 의 비밀 `.env` 차단은 서브에이전트의 도구 호출에도 그대로 적용된다.
- **.claude/skills/** — 필요할 때만 로드되는 반복 절차서. `plan-feature` 가 들어 있어
  기능을 새로 만들거나 고칠 때(버그수정·리팩터 포함) 쓴다 — 큰 작업은 작게 쪼개 계획(PLAN.md)·진행상태(TASKS.md)로 한 번에 하나씩, 작은 변경은 바로 고친 뒤 동작을 같이 확인한다.
  `make-rule` 도 들어 있어 "앞으로 ○○하게 해줘"라고 하면 적용 범위를 알아서 판단해 규칙 파일을 만든다.
- **.claude/rules/** — 특정 작업에만 적용할 세부 규칙 자리. 여기 둔 규칙 파일은 **자동으로 인식된다** —
  맨 위 `paths` 에 적은 경로를 건드릴 때만 켜지고, `paths` 가 없으면 항상 켜진다(항상 토큰을 쓰므로 진짜 전역 규칙만).
  직접 만들 필요 없이 "앞으로 ○○하게 해줘"라고 하면 `make-rule` 스킬이 만들어 준다.
  폴더는 처음엔 없고, 첫 규칙을 만들 때 생긴다.
- **.gitignore** — git이 무시할 파일 목록(설치물, 비밀키, 빌드 산출물 등). 견본 파일(`.env.example`)은 공유해도 안전하므로 예외로 둔다.
- **HARNESS.md** — (이 파일) 사람이 읽는 구조 안내서. Claude가 자동으로 읽지 않는다.

## 왜 스택·MCP 같은 건 비워 뒀나
기술 스택, 사용하는 MCP 등은 **프로젝트마다 다르다.** 기본 골격에 미리 박아두면
새 프로젝트에서 오히려 방해가 된다. 그래서 변하지 않는 규칙만 채워 두고,
변하는 것은 **빈 칸(SPEC.md)** 으로 두거나 **아예 만들지 않는다(.mcp.json 등).**
필요한 프로젝트에서 그때 직접 추가한다.

## 다음에 할 일
1. `SPEC.md` 를 확인한다 — 신규면 직접 채우고, 기존 프로젝트면 `/womc` 가 코드에서 뽑아 채운 초안을 검토·수정한다.
2. 기능을 만들거나 고칠 땐 "○○ 만들자 / ○○ 고쳐줘"라고 한다. `plan-feature` 스킬이 작업 크기를 가늠해
   큰 작업이면 계획(PLAN.md)·진행상태(TASKS.md)를 그때 만들어 한 번에 하나씩, 작은 변경이면 바로 처리하며,
   조사·설계·구현·검증·검토를 전문 서브에이전트에 (필요하면 병렬로) 맡긴다.
3. 한 기능이 동작하는 걸 확인하면 커밋해 되돌릴 지점을 남긴다 — "저장(커밋)해 둘까요?"라고 물으면 Claude가 대신 `git add`·`git commit` 해 준다(`push` 는 자동으로 하지 않는다).

> PLAN.md·TASKS.md 는 처음엔 없다. 필요한 순간에만 생겨 평소 컨텍스트를 가볍게 둔다.
> 나중에 womc 플러그인의 새 버전이 나오면 `/womc update` 로 골격만 갱신할 수 있다(직접 채운 파일은 보존된다).
```

### 5) `.claude/` 하위 폴더와 하네스 파일들
아래 파일들을 각각 그대로 만든다.
(안내용 README 는 `.claude/` 하위에 만들지 않는다 — `.claude/rules/` 의 `.md` 는 **매 세션 자동 로드**되고,
`.claude/agents/` 의 `.md` 는 서브에이전트 정의로 스캔되므로, 실제 동작 파일만 둔다.

## 2. 조사 (필요하면 — explore 위임)
기능이 어디에 어떻게 얽히는지 모르면 `explore` 서브에이전트에게 조사를 맡긴다.
서로 다른 영역을 봐야 하면 **한 메시지에서 explore 여러 개를 병렬로** 띄운다. 결론(`파일경로:줄번호`)만 돌아와 메인이 가벼워진다.

## 3. 설계 (비자명하면 — plan 위임)
여러 파일이 얽히거나 순서가 애매한 기능이면 `plan` 서브에이전트에게 설계도를 맡긴다(2의 조사 결과를 함께 넘긴다).
돌아온 단계별 계획을 `PLAN.md`/`TASKS.md` 에 반영한다. 각 단계는 파일 1~2개 수준으로 쪼갠다.
(간단한 기능이면 이 단계를 건너뛰고 바로 쪼개도 된다.)

## 4. 구현 (implement 위임)
각 단계를 `implement` 서브에이전트에 맡긴다 — 단계 명세 + 손댈 파일 경로 + 따라야 할 패턴(`파일경로:줄번호`)을 함께 넘긴다.
서로 다른 파일을 건드리는 독립 단계는 **여러 개를 병렬로** 맡길 수 있다. 같은 파일을 건드리는 단계는 순서대로.
돌아온 요약으로 진행 표기를 `TASKS.md` 에 갱신한다(`[~]` → `[x]`). 코드는 implement 가 파일에 직접 쓰고, 메인은 결과를 조립·확인한다.

## 5. 검증 (verify 위임)
구현이 끝나면 확인 방법이 이미 기록돼 있는지 본다(`TASKS.md` 나 `SPEC.md`).
- **기록돼 있으면**: `verify` 서브에이전트에게 위임해 통과/실패만 받는다. 테스트·실행 로그가 이 대화에 쌓이지 않는다.
- **처음이면**: 프로젝트를 살펴 확인 방법을 1~2가지 골라, 쉬운 말로 먼저 제안한다.
  (예: "터미널에서 이 명령 하나를 돌려 '통과'가 뜨는지 보면 됩니다") 사용자는 제안 중 하나를 고르기만 하면 된다.
  확인 방법을 알아내는 일을 사용자에게 떠넘기지 않는다. (실행 명령·테스트는 프로젝트마다 다르므로 미리 박지 않는다.)

확인되면 그 방법을 `TASKS.md` 에 한 줄로 남긴다 — 다음부터는 묻지 않고 `verify` 에게 위임한다.

## 6. 검토 (review 위임 — 선택)
한 단계나 기능이 끝나면 `review` 서브에이전트에게 바뀐 부분(diff)을 맡겨 버그·개선점을 받는다.
심각한 것부터 메인이 반영한다(고치는 건 메인 몫 — review 는 지적만 한다).

## 7. 다음으로
한 단계가 동작하는 걸 확인한 뒤에만 다음 단계로 간다.
동작을 확인했으면 사용자에게 커밋을 권한다 — 되돌릴 지점을 남기는 것이다. 사용자는 코딩을 모르므로 "지금까지 한 걸 저장(커밋)해 둘까요?"처럼 쉬운 말로 묻고, 좋다고 하면 메인이 대신 `git add`·`git commit` 한다. `push` 는 자동으로 하지 않는다.
한 기능이 다 끝나면 새 세션으로 시작하길 권한다(진행상태는 PLAN.md/TASKS.md 에 남아 있다).
작업 중간에 대화가 무거워지면 사용자에게 `/compact` 를 제안한다.
```

`.claude/skills/make-rule/SKILL.md` (실제 동작하는 스킬 — 사용자는 의도만, 기술 변환은 스킬이 강제)
```markdown
---
name: make-rule
description: 사용자가 "○○를 규칙으로 정해줘", "앞으로 △△하게 해줘", "자꾸 ○○하지 말고 ~~하게 해줘"처럼 앞으로 계속 지킬 규칙을 원하면 사용한다. 사용자는 평소 말로 의도만 설명한다. paths 스코프·파일 형식 같은 기술적인 건 이 스킬이 알아서 채운다.
---
# 규칙 만들기

사용자가 "앞으로 ○○하게 해줘" 같이 반복해서 지킬 규칙을 원하면 이 절차를 따른다.
사용자는 코딩 용어(paths, 스코프, 폴더 경로 등)를 모른다고 전제한다. 의도만 듣고, 기술적인 변환은 전부 이 스킬이 한다.

## 0. 규칙 의도 정리
사용자가 원하는 걸 규칙 한 문장으로 바꿔 보여주고 맞는지 확인한다.
예: "색이랑 간격 제멋대로 바꾸지 마" → "색과 간격은 정해둔 값만 쓴다"

## 1. 적용 범위 판단 (핵심)
이 규칙이 프로젝트 어디서 켜져야 하는지 직접 정한다. 사용자에게 폴더 경로를 묻지 않는다.
- 먼저 프로젝트 구조를 살핀다(Glob 등으로 폴더·파일 종류 파악).
- 규칙이 특정 영역에만 해당하면(예: 디자인 규칙 → 스타일/CSS, 저장 규칙 → 데이터 관련 파일) 그 영역만 스코프한다.
- 폴더가 역할별로 안 나뉘어 있으면 파일 종류로 스코프한다(예: 스타일 규칙이면 `**/*.css`).
- 정말 프로젝트 전체에 해당하면(예: "모든 설명은 한국어로") 범위를 전체로 둔다.
- 어느 쪽인지 애매하면 용어 없이 한 줄로 되묻는다.
  예: "이 규칙은 화면 디자인할 때만 지키면 되나요, 아니면 어디서나 지켜야 하나요?"

## 2. 파일 만들기
`.claude/rules/<알기쉬운이름>.md` 를 만든다. 같은 이름이 있으면 덮어쓰지 말고 알린다.

- 특정 영역만 적용이면, 맨 위에 paths 를 적어 그 경로에서만 켜지게 한다:
~~~markdown
---
paths:
  - "<해당 경로>/**"
---
<규칙 문장>
~~~
이렇게 하면 그 경로의 파일을 건드릴 때만 규칙이 켜지고, 다른 작업에선 꺼져 있어 토큰을 아낀다.

- 프로젝트 전체 적용이면, paths 없이 규칙 문장만 적는다(항상 켜진다). 전체 적용은 항상 토큰을 쓰므로, 정말 전역일 때만 이렇게 한다.
~~~markdown
<규칙 문장>
~~~

`.claude/rules/` 안의 파일은 CLAUDE.md 에 따로 import 하지 않아도 자동으로 인식된다.
(paths 가 있으면 그 경로를 건드릴 때, 없으면 항상 켜진다.)

무슨 규칙을 만들었고 언제 켜지는지, 용어 없이 한국어로 짧게 알린다.
예: "디자인 규칙을 만들었어요. 스타일을 손볼 때만 자동으로 적용되고, 다른 작업엔 영향 없습니다."

### 6) `.claude/settings.json` (하네스 — 권한 기본값)
아래 내용을 그대로 쓴다.
- `deny` 로 비밀키 파일(`.env` 와 그 변형)을 **하위 폴더 것까지**(`**/.env`) Claude가 읽지 못하게 막는다. (.gitignore 의 비밀키 제외와 같은 정책)
  단, 비밀이 없는 견본 파일(`.env.example`)은 읽을 수 있어야 하므로 — Claude가 "어떤 설정값이 필요한지" 파악하는 데 쓴다 —
  `.env.*` 처럼 통째로 막지 않고 **비밀이 들어가는 이름만 골라서** 막는다. (deny 는 allow 보다 우선이라, 통째로 막으면 견본만 예외로 풀 방법이 없다.
  흔한 이름 기준이므로 특이한 이름의 비밀 파일까지 다 잡지는 못한다 — 진짜 중요한 비밀은 프로젝트 폴더 밖에 둔다.)
  또한 이 차단은 파일 읽기 도구(Read)에만 걸린다. 터미널 명령으로 읽는 것까지 다 막지는 못한다.
- `git status` 같은 Bash 읽기 전용 명령은 Claude Code 가 **기본으로 프롬프트 없이 허용**하므로 적지 않는다.
  Windows(PowerShell)에서 자주 쓰는 읽기 전용 명령만 미리 허용해 둔다. (Mac/Linux 에서는 PowerShell 항목이 조용히 무시된다 — 해는 없다.)
- `push`·`reset` 같이 되돌리기 어려운 명령은 일부러 넣지 않는다 — 그건 매번 확인받는 게 맞다.

```json
{
  "permissions": {
    "allow": [
      "PowerShell(git status:*)",
      "PowerShell(git diff:*)",
      "PowerShell(git log:*)",
      "PowerShell(Get-ChildItem:*)"
    ],
    "deny": [
      "Read(**/.env)",
      "Read(**/.env.local)",
      "Read(**/.env.*.local)",
      "Read(**/.env.development)",
      "Read(**/.env.production)",
      "Read(**/.env.test)",
      "Read(**/.env.staging)"
    ]
  }
}
```

## 기존 프로젝트 온보딩 (해당할 때만)
파일 생성을 마친 뒤, **폴더에 기존 코드가 있거나 이미 사용자가 쓴 `CLAUDE.md`·`.claude/settings.json` 이 있으면** 아래 온보딩을 진행한다. (신규 빈 폴더면 이 절 전체를 건너뛴다.)
아래 각 단계는 **자기 조건이 맞을 때만** 켜진다 — SPEC 초안·관례 캡처·구조 제안은 *기존 코드가 있을 때*, CLAUDE.md·설정 조율은 *그 파일이 이미 있을 때*(코드가 거의 없어도).
어떤 단계든 **기존 코드·파일 내용은 지우거나 덮어쓰지 않는다** — 추가·병합·제안만 한다.

- **기존 코드 판단**: `Glob` 로 매니페스트(`package.json`·`pyproject.toml`·`go.mod`·`Cargo.toml`·`pom.xml`·`Gemfile`·`composer.json` 등)나 소스 폴더(`src/` 등)·실제 코드 파일이 보이면 기존 프로젝트다. 방금 만든 골격 파일·`.git`·`node_modules` 는 신호에서 뺀다.

### 1) SPEC 초안 (이번에 `SPEC.md` 를 새로 만들었을 때만)
`SPEC.md` 가 이미 있었으면 이 단계는 건너뛴다.
- 조사는 `explore` 서브에이전트에 위임한다(직접 코드를 읽지 않아 이 대화가 가벼워진다). 영역이 여럿이면 한 메시지에서 병렬로 띄운다 — 예: ① 스택·빌드/실행 ② 핵심 기능·진입점 ③ 다루는 데이터·외부 연동. 각 `explore` 는 `파일경로:줄번호` 결론만 돌려준다.
- 돌아온 결론으로 **네(메인)가** `SPEC.md` 의 6섹션을 채운다. 확실치 않은 칸은 비워 두거나 "확인 필요"로 남긴다. 코드는 구현하지 않는다.

### 2) 기존 `CLAUDE.md`·설정 조율 (이미 있을 때만)
프로젝트에 이미 **비-womc `CLAUDE.md`** 가 있으면(첫 줄이 골격의 `# 작업 규칙 (모든 프로젝트 공통 · 불변)` 이 아니면) 건너뛰지 말고 조율한다:
- 기존 파일을 읽고 **맨 끝에** 아래처럼 표시된 구획을 덧붙인다 — 사용자 내용은 한 줄도 바꾸지 않는다. 이미 `@SPEC.md` import 나 `womc:` 구획이 있으면 다시 넣지 않는다.
~~~markdown

<!-- womc:begin — 이 구획만 /womc update 가 관리. 위쪽 사용자 내용은 건드리지 않음 -->
## 나에 대해
(여기에 이 문서 CLAUDE.md 골격의 **본문 전체**를 맨 위 H1 제목(`# 작업 규칙 …`)만 빼고 그대로 넣는다 — '나에 대해'·'코드 제공 방식'·'설명 방식'·'적극 위임'·'프로젝트 상세'(`@SPEC.md`)의 5개 절 모두. 신규 프로젝트가 받는 규칙과 똑같이 맞추기 위해서다.)
<!-- womc:end -->
~~~
- 기존 `.claude/settings.json` 이 있으면 통째로 덮지 말고 **갱신 모드와 같은 방식**(사용자 항목 보존 + 이 문서의 기본 allow/deny 만 추가)으로 병합한다.
- 사용자의 기존 CLAUDE.md 가 언어·수정 방식 등 같은 주제를 이미 정해 뒀으면, 겹치는 부분을 마지막 안내에서 한 줄로 짚어 주되 사용자 문장은 지우지 않는다.
- 무엇을 병합했는지 마지막 안내에서 한 줄로 알린다.

### 3) 코드 관례를 규칙으로 캡처 (제안 후에만)
기존 코드에 일관된 관례(명명·폴더 역할·포매팅·테스트 방식 등)가 보이면, 앞으로 유지보수가 그걸 따르도록 `.claude/rules/` 로 캡처할지 **먼저 한 줄로 제안**한다. 좋다고 하면:
- 조사는 `explore` 에 위임해 관례를 뽑는다(`파일경로:줄번호` 근거).
- 영역이 뚜렷하면 `paths` 로 스코프한 규칙 파일을 만든다(예: 스타일 관례 → 맨 위 `paths: ["**/*.css"]`). 애매하면 만들지 않는다.
- 확실한 관례만 옮긴다 — 추측으로 규칙을 만들지 않는다. (형식은 `make-rule` 스킬과 같다.)

### 4) 구조 정리는 제안만 → `plan-feature` 로 넘김
코드·폴더 구조가 어질러져 보여도 **`/womc` 는 코드를 옮기거나 고치지 않는다.** 마지막 안내에서 한 줄로 제안만 한다:
- 예: "구조를 정리하고 싶으면 '구조 정리해줘'라고 하세요 — `plan-feature` 가 단계별로, 되돌릴 수 있게(커밋) 진행합니다."
- 사용자가 원하면 그때 `plan-feature` 리팩터 경로로 넘어간다. `/womc` 는 감지·제안까지만 한다.

온보딩을 마치면 무엇을 했는지(SPEC 초안·병합·캡처·구조 제안 중 실제로 한 것)를 한국어로 한 줄씩 알리고 "맞는지 검토하고 고쳐 주세요"로 맺는다.

## 생성이 끝나면
먼저, 골격 생성을 모두 마친 뒤 마무리 안내 맨 앞에 아래 WOMC 로고 아트를 코드블록(```)으로 감싸 이 모양 그대로 한 번 출력한다.
(일부 파일을 건너뛴 경우에도 아트는 동일하게 맨 앞에 한 번 넣는다.) 아트와 태그라인을 출력하고 한 줄 비운 뒤, 아래 한국어 안내로 이어간다.

```
 __        _____  __  __  ____
 \ \      / / _ \|  \/  |/ ___|
  \ \ /\ / / | | | |\/| | |
   \ V  V /| |_| | |  | | |___
    \_/\_/  \___/|_|  |_|\____|

           works on my claude
```

그다음 한국어로 짧게 다음 단계를 안내한다. **신규 빈 폴더였는지, 기존 프로젝트였는지에 따라 1)번을 다르게** 안내한다. 예시:

> ✅ 기본 골격을 만들었어요.
> 다음 순서로 진행하면 됩니다.

## 갱신 모드 (`/womc update`)
인자에 `update` 가 있을 때만 이 절차를 수행한다. 원칙: **사용자가 채운 내용은 절대 건드리지 않고, 불변 골격만 이 문서의 최신 내용으로 교체한다.**

1. 아래 불변 골격 파일들을 이 문서에 적힌 내용 그대로 **덮어쓴다**(없으면 새로 만든다):
   - `CLAUDE.md` — 세 경우로 나뉜다: ⓐ 첫 줄이 골격 그대로(`# 작업 규칙 (모든 프로젝트 공통 · 불변)`)면 통째로 덮어쓴다. ⓑ **온보딩으로 사용자 파일에 병합된 형태(`<!-- womc:begin -->`…`<!-- womc:end -->` 구획이 있는 경우)면 그 구획 안만 최신 내용으로 교체**하고 나머지 사용자 내용은 그대로 둔다. ⓒ 둘 다 아니면(사용자가 쓴 CLAUDE.md 인데 온보딩 병합을 안 거친 경우) **통째로 덮지 말고**, 위 '기존 프로젝트 온보딩'의 CLAUDE.md 병합 절차대로 `womc:begin/end` 구획을 맨 끝에 덧붙인다(사용자 내용 보존).
   - `HARNESS.md`
   - `.claude/agents/explore.md` · `.claude/agents/plan.md` · `.claude/agents/implement.md` · `.claude/agents/verify.md` · `.claude/agents/review.md`
   - `.claude/skills/plan-feature/SKILL.md` · `.claude/skills/make-rule/SKILL.md`
2. `.claude/settings.json` 은 통째로 덮어쓰지 않는다. 기존 파일을 먼저 읽고,
   **사용자가 직접 추가한 항목(allow/deny 등 이 문서의 기본값에 없는 것)은 그대로 둔 채** 기본값 항목만 최신으로 맞춰 다시 쓴다.
   (기존 파일이 없으면 기본값으로 새로 만든다. 예전 기본값이던 `Read(**/.env.*)` 가 있으면 위의 새 deny 목록으로 교체한다 — 견본 `.env.example` 을 읽을 수 있게 하기 위해서다.)
3. 다음은 **절대 덮어쓰지도 지우지도 않는다**: `SPEC.md` · `PLAN.md` · `TASKS.md` · `.claude/rules/` 전체 · `.gitignore` ·
   그 외 이 문서에 없는 모든 파일. (`.gitignore` 는 사용자가 항목을 추가했을 수 있어 갱신 대상에서 뺀다.)
4. 예전 버전이 만든 `README.md` 가 있으면(첫 줄이 `# 프로젝트 구조 안내` 인 경우) 지우지 말고,
   "예전 안내 파일인데 이제 `HARNESS.md` 로 대체됐어요. 직접 지우거나 프로젝트 소개용으로 바꿔 쓰시면 됩니다"라고 안내만 한다.
   (첫 줄이 다르면 사용자가 만든 README 이므로 아무 말도 하지 않는다.)
5. 끝나면 생성 모드와 동일하게 WOMC 로고 아트를 맨 앞에 출력한 뒤, 무엇을 교체했고 무엇을 보존했는지 한국어로 짧게 알린다.
