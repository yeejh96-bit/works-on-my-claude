description: 새 프로젝트 폴더에 항상 동일한 기본 골격(컨텍스트 엔지니어링 + 하네스 세팅)을 생성한다
allowed-tools: Write, Read, Glob

너는 지금 현재 작업 폴더에 "프로젝트 기본 골격"을 생성하는 작업만 한다.
아래 규칙을 반드시 지켜라.

## 절대 규칙
- 아래에 적힌 파일/폴더만, 적힌 내용 그대로 생성한다. (매 프로젝트 항상 동일)
- 프로젝트마다 달라지는 것(기술 스택, 사용하는 MCP 등)은 채우지 않는다. 빈 틀만 둔다.
- `.mcp.json` 은 만들지 않는다.
- SPEC 자동 작성, 코드 구현은 하지 않는다. 구조 생성까지만 한다.
- 생성하기 전에 먼저 `Glob` 로 현재 폴더에 같은 이름 파일이 이미 있는지 확인한다.
  이미 있는 파일은 **덮어쓰지 말고 건너뛰며**, 어떤 파일이 이미 있었는지 마지막 안내에서 알려준다.

## 생성할 파일과 내용

### 1) `CLAUDE.md`
아래 내용을 그대로 쓴다.

```markdown
# 작업 규칙 (모든 프로젝트 공통 · 불변)

이 파일은 매 세션 자동으로 로드된다. 아래 규칙은 프로젝트와 상관없이 항상 적용된다.

## 나에 대해
- 나는 코딩을 모른다. 전문 용어를 최소화하고, 필요한 건 풀어서 설명한다.

## 코드 제공 방식
- 코드 수정은 내가 직접 한다. 너는 복사·붙여넣기 할 필요 없다. (바뀐 부분만 정확히 고친다)
- 보여줄 때는 무엇을 왜 바꿨는지 한국어로 설명하고, 필요한 만큼만 보여준다. (작은 파일이면 전체, 큰 파일이면 바뀐 부분 위주 — 토큰 낭비를 막는다)
- 어떤 파일인지 파일 경로를 함께 적는다.
- 파일을 새로 만들거나 바꾸면, 그다음 무엇을 하면 되는지 한 줄로 알려준다.

## 설명 방식
- 모든 설명은 한국어로 한다.

## 컨텍스트·토큰 아끼기 (하네스 습관)
- 여러 파일을 뒤져야 하는 무거운 조사는 `explore` 서브에이전트에게 맡긴다. 조사는 별도 컨텍스트에서 끝나고 결론만 돌아오므로, 메인 대화에 파일 내용이 쌓이지 않는다.
- 한 번에 한 기능만 진행한다. 큰 작업을 한꺼번에 벌이지 않는다.
- 세션이 길어지면 `/compact` 로 대화를 정리한 뒤 이어간다.

## 프로젝트 상세
이 프로젝트의 구체적인 내용(스택, 무엇을 만드는지 등)은 아래 파일에 있다.

@SPEC.md
```

### 3) `README.md`
아래 내용을 그대로 쓴다.

```markdown
# 프로젝트 구조 안내

이 폴더는 Claude Code가 일을 잘하도록 **컨텍스트 엔지니어링**과 **하네스 세팅**을
미리 갖춘 구조다. 각 파일이 맡은 역할은 다음과 같다.

## 파일별 역할
- **CLAUDE.md** — 프로젝트와 상관없이 항상 적용되는 불변 작업 규칙.
  Claude Code를 켜면 **매 세션 자동으로 로드**된다. (예: 한국어로 설명, 수정은 직접 하고 필요한 만큼만 보여줌)
- **SPEC.md** — 이 프로젝트만의 상세 내용(스택, 만들 것, 기능 등). CLAUDE.md 끝에서
  `@SPEC.md` 로 import 되어 **CLAUDE.md에 딸려 함께 로드**된다. 처음엔 빈 템플릿이며 직접 채운다.
- **.claude/settings.json** — 하네스(작업 환경) 기본값. 어디서나 안전한 읽기 전용 명령
  (`git status`, `git diff`, `git log`, 파일 목록)을 미리 허용해, 매번 허용 프롬프트가 뜨는 걸 줄인다.
- **.claude/agents/** — 무거운 일을 별도 컨텍스트에서 처리하는 서브에이전트. `explore` 가 들어 있어
  여러 파일을 뒤지는 조사를 메인 대화 밖에서 끝내고 결론만 가져오며, `verify` 가 들어 있어
  "이게 되는지" 동작 검증을 격리된 컨텍스트에서 끝내고 통과/실패만 가져온다(토큰 절약).
- **.claude/skills/** — 필요할 때만 로드되는 반복 절차서. `plan-feature` 가 들어 있어
  새 기능을 작게 쪼개 한 번에 하나씩 구현하고, 계획(PLAN.md)·진행상태(TASKS.md)를 그때 만들어 관리하며, 끝나면 동작을 같이 확인한다.
  `make-rule` 도 들어 있어 "앞으로 ○○하게 해줘"라고 하면 적용 범위를 알아서 판단해 규칙 파일을 만든다.
- **.claude/rules/** — 특정 작업에만 적용할 세부 규칙 자리. 자동 로드되지 않으며 `@import` 로 켠다.
- **.gitignore** — git이 무시할 파일 목록(설치물, 비밀키, 빌드 산출물 등).

## 왜 스택·MCP 같은 건 비워 뒀나
기술 스택, 사용하는 MCP 등은 **프로젝트마다 다르다.** 기본 골격에 미리 박아두면
새 프로젝트에서 오히려 방해가 된다. 그래서 변하지 않는 규칙만 채워 두고,
변하는 것은 **빈 칸(SPEC.md)** 으로 두거나 **아예 만들지 않는다(.mcp.json 등).**
필요한 프로젝트에서 그때 직접 추가한다.

## 다음에 할 일
1. `SPEC.md` 를 채운다.
2. 기능을 만들 땐 "○○ 기능 만들자"라고 한다. `plan-feature` 스킬이 계획(PLAN.md)·진행상태(TASKS.md)를
   그때 만들고, 한 번에 하나씩 구현한 뒤 동작을 같이 확인한다.

> PLAN.md·TASKS.md 는 처음엔 없다. 필요한 순간에만 생겨 평소 컨텍스트를 가볍게 둔다.
```

`.claude/agents/README.md`
```markdown
# agents
이 프로젝트 전용 서브에이전트를 두는 곳이다.
무거운 조사를 별도 컨텍스트에서 처리해 메인 대화의 토큰을 아낀다.
- `explore.md` — 코드 조사 전용. "어디 있지 / 어떻게 동작하지"를 물으면 여기로 위임된다.
- `verify.md` — 구현 검증 전용. "이게 되는지 확인해줘"처럼 동작 확인을 시키면 여기로 위임된다.

```

`.claude/agents/explore.md` (실제 동작하는 조사 전용 서브에이전트)
```markdown
---
name: explore
description: 코드베이스에서 "이게 어디 있지 / 어떻게 동작하지"를 조사할 때 사용한다. 여러 파일을 읽어야 하는 무거운 조사를 별도 컨텍스트에서 처리하고 결론만 돌려줘, 메인 대화의 토큰을 아낀다.
tools: Read, Grep, Glob
---
너는 코드 조사 전용 에이전트다. 메인 대화의 컨텍스트를 아끼는 것이 존재 이유다.

규칙:
- 요청받은 것만 조사하고, 끝나면 결론만 짧게 돌려준다.
- 파일 내용을 통째로 붙여넣지 않는다. 대신 `파일경로:줄번호` 로 위치만 가리킨다.
- 코드를 수정하지 않는다. 읽기·검색만 한다.
- 확실하지 않으면 "확인 못 함"이라고 분명히 적는다.

출력 형식:
1) 한 줄 요약
2) 핵심 발견 (불릿마다 `파일경로:줄번호`)
3) 다음에 보면 좋을 곳 (선택)
```

`.claude/agents/verify.md` (실제 동작하는 검증 전용 서브에이전트 — 읽기·실행만, 수정 안 함)
```markdown
---
name: verify
description: 방금 구현한 기능이 실제로 동작하는지 검증할 때 사용한다. "이게 되는지 확인해줘", "테스트 돌려서 잘 되는지 봐줘", "동작 검증해줘"처럼 구현 결과 확인을 요청하면 여기로 위임된다. 테스트·실행으로 확인하고 통과/실패만 돌려준다. 코드는 고치지 않는다.
tools: Read, Grep, Glob, Bash
---
너는 구현 검증 전용 에이전트다. 메인 대화의 컨텍스트를 아끼고, 위임된 검증만 하는 것이 존재 이유다.

규칙:
- 위임받은 것만 검증하고, 끝나면 결론만 돌려준다.
- 코드를 수정하지 않는다. 문제를 찾으면 무엇이 왜 실패했는지 보고만 하고, 고치는 건 메인이 한다.
- 검증 방법(실행 명령·테스트)은 프로젝트에 맞게 고른다. 확실치 않으면 "이 방법으로 확인했다"를 함께 적는다.
- 로그·출력을 통째로 붙여넣지 않는다. 실패한 핵심 줄과 `파일경로:줄번호` 로 위치만 가리킨다.
- 확실하지 않으면 "확인 못 함"이라고 분명히 적는다.
- 모든 보고는 한국어로 한다.

출력 형식:
1) 통과 / 실패 (한 줄)
2) 무엇을 어떻게 확인했는지 (실행한 명령과 핵심 결과)
3) 실패 시: 어디가 왜 문제인지 (`파일경로:줄번호`)
```

# skills
이 프로젝트 전용 스킬(반복 절차서)을 두는 곳이다. 필요할 때만 로드돼 컨텍스트를 아낀다.
- `plan-feature/SKILL.md` — 새 기능을 작게 쪼개 한 번에 하나씩 구현하는 절차.
- `make-rule/SKILL.md` — "앞으로 ○○하게 해줘"라고 하면, 적용 범위를 알아서 판단해 규칙 파일을 만드는 절차.
직접 추가하려면 `스킬이름/SKILL.md` 형식으로 폴더를 만든다.
```

`.claude/skills/plan-feature/SKILL.md` (실제 동작하는 스킬 — 계획·진행상태·검증을 한 흐름으로 처리)
```markdown
---
name: plan-feature
description: 새 기능을 하나 추가할 때 사용한다. 큰 작업을 작은 단계로 쪼개 한 번에 하나씩 구현하고, 진행 상황을 PLAN.md/TASKS.md 에 남기며, 끝나면 동작을 함께 확인한다.
---
# 한 기능 추가하기

사용자가 "○○ 기능 만들자"고 하면 이 절차를 따른다.

## 0. 상태 파일 준비
`PLAN.md` / `TASKS.md` 가 없으면 아래 빈 템플릿으로 만든다(내용은 채우지 않는다). 이미 있으면 읽어 현재 상태를 파악한다.
이 두 파일은 CLAUDE.md 에 import 하지 않는다 — 이 스킬이 작업 중에만 읽고 갱신한다(평소 컨텍스트를 차지하지 않게).

`PLAN.md` (없을 때만 생성):
~~~markdown
# 구현 계획 (PLAN)

> SPEC을 "무엇을 어떤 순서로 만들지"로 옮긴 큰 그림이다.
> 자잘한 진행 상태는 TASKS.md 가 맡는다. 여기는 순서/마일스톤만.

## 만드는 순서
1. (아직 비어 있음 — SPEC.md 를 채운 뒤 큰 덩어리 순서를 적는다)

## 지금 집중하는 단계
(아직 없음)

## 나중에 / 안 할 것
(아직 없음)
~~~

`TASKS.md` (없을 때만 생성):
~~~markdown
# 작업 목록 (TASKS)

> PLAN의 한 단계를 실제 작업으로 쪼갠 체크리스트다.
> 표기: `[ ]` 안 함 · `[~]` 하는 중 · `[x]` 끝남

## 지금 하는 일
(아직 없음)

## 할 일 (순서대로)
- [ ] (아직 비어 있음)

## 끝난 일
(아직 없음)
~~~

## 1. SPEC 확인
`SPEC.md` 에 이 기능이 있는지 본다. 없으면 한 줄로 추가를 제안한다.

## 2. 작게 쪼개기
기능을 3단계 이하로 나눠 `TASKS.md` 의 "할 일"에 적고 보여준다. 각 단계는 파일 1~2개 수준이어야 한다.

## 3. 한 단계만 구현
한 번에 한 단계만. 수정은 직접 하고, 필요한 만큼만 보여준다(CLAUDE.md 규칙).
진행 표기를 `TASKS.md` 에 갱신한다(`[~]` → `[x]`).

## 4. 검증 (언어 비종속)
구현이 끝나면 "이게 되는지 어떻게 확인하면 될까요?"를 사용자에게 묻고 함께 확인한다.
(실행 명령·테스트 방법은 프로젝트마다 다르므로 미리 박지 않는다.)
확인되면 그 방법을 `SPEC.md` 나 `TASKS.md` 에 한 줄로 남긴다.

## 5. 다음으로
한 단계가 동작하는 걸 확인한 뒤에만 다음 단계로 간다.
세션이 길어지면 `/compact` 로 정리하고 이어간다.
```

`.claude/skills/make-rule/SKILL.md` (실제 동작하는 스킬 — 사용자는 의도만, 기술 변환은 스킬이 강제)
```markdown
---
name: make-rule
description: 사용자가 "○○를 규칙으로 정해줘", "앞으로 △△하게 해줘", "자꾸 ○○하지 말고 ~~하게 해줘"처럼 앞으로 계속 지킬 규칙을 원하면 사용한다. 사용자는 평소 말로 의도만 설명한다. paths 스코프·파일 형식 같은 기술적인 건 이 스킬이 알아서 채운다.
---
# 규칙 만들기

사용자가 "앞으로 ○○하게 해줘" 같이 반복해서 지킬 규칙을 원하면 이 절차를 따른다.
사용자는 코딩 용어(paths, 스코프, 폴더 경로 등)를 모른다고 전제한다. 의도만 듣고, 기술적인 변환은 전부 이 스킬이 한다.

## 0. 규칙 의도 정리
사용자가 원하는 걸 규칙 한 문장으로 바꿔 보여주고 맞는지 확인한다.
예: "색이랑 간격 제멋대로 바꾸지 마" → "색과 간격은 정해둔 값만 쓴다"

## 1. 적용 범위 판단 (핵심)
이 규칙이 프로젝트 어디서 켜져야 하는지 직접 정한다. 사용자에게 폴더 경로를 묻지 않는다.
- 먼저 프로젝트 구조를 살핀다(Glob 등으로 폴더·파일 종류 파악).
- 규칙이 특정 영역에만 해당하면(예: 디자인 규칙 → 스타일/CSS, 저장 규칙 → 데이터 관련 파일) 그 영역만 스코프한다.
- 폴더가 역할별로 안 나뉘어 있으면 파일 종류로 스코프한다(예: 스타일 규칙이면 `**/*.css`).
- 정말 프로젝트 전체에 해당하면(예: "모든 설명은 한국어로") 범위를 전체로 둔다.
- 어느 쪽인지 애매하면 용어 없이 한 줄로 되묻는다.
  예: "이 규칙은 화면 디자인할 때만 지키면 되나요, 아니면 어디서나 지켜야 하나요?"

## 2. 파일 만들기
`.claude/rules/<알기쉬운이름>.md` 를 만든다. 같은 이름이 있으면 덮어쓰지 말고 알린다.

- 특정 영역만 적용이면, 맨 위에 paths 를 적어 그 경로에서만 켜지게 한다:
~~~markdown
---
paths:
  - "<해당 경로>/**"
---
<규칙 문장>
~~~
이렇게 하면 그 경로의 파일을 건드릴 때만 규칙이 켜지고, 다른 작업에선 꺼져 있어 토큰을 아낀다.

- 프로젝트 전체 적용이면, paths 없이 규칙 문장만 적는다(항상 켜진다). 전체 적용은 항상 토큰을 쓰므로, 정말 전역일 때만 이렇게 한다.
~~~markdown
<규칙 문장>
~~~

`.claude/rules/` 안의 파일은 CLAUDE.md 에 따로 import 하지 않아도 자동으로 인식된다.
(paths 가 있으면 그 경로를 건드릴 때, 없으면 항상 켜진다.)

## 3. 마무리 안내
무슨 규칙을 만들었고 언제 켜지는지, 용어 없이 한국어로 짧게 알린다.
예: "디자인 규칙을 만들었어요. 스타일을 손볼 때만 자동으로 적용되고, 다른 작업엔 영향 없습니다."
```

`.claude/rules/README.md`
```markdown
# rules
특정 파일이나 작업에만 적용할 세부 규칙 문서를 두는 곳이다.
여기 둔 문서는 자동으로 로드되지 않는다. 필요할 때만 켠다.

## 쓰는 법 (필요해지면)
1. 아래 틀 중 필요한 걸 복사한다.
2. `.claude/rules/coding.md` 처럼 파일로 저장한다.
3. `CLAUDE.md` 맨 끝에 `@.claude/rules/coding.md` 한 줄을 더한다.
지금은 아무 규칙도 켜져 있지 않다(빈 상태가 기본이다).

## 복사해서 쓰는 최소 틀

### coding.md
~~~markdown
# 코딩 규칙
- 한 파일은 한 가지 일만 한다.
- 이름은 풀어서 짓는다(약어 피하기).
- 새 라이브러리는 추가 전에 한 줄로 이유를 묻는다.
~~~

### ui.md
~~~markdown
# UI 규칙
- 색·간격은 정해둔 값만 쓴다(즉흥 값 금지).
- 버튼/입력 등 같은 요소는 한 군데서 정의해 재사용한다.
- 모바일 화면을 먼저 맞춘다.
~~~

### architecture.md
~~~markdown
# 구조 규칙
- 폴더는 "역할"로 나눈다(화면 / 데이터 / 공용).
- 한 폴더가 다른 폴더를 너무 많이 알지 않게 한다.
- 큰 결정(폴더 구조·데이터 흐름)은 바꾸기 전에 먼저 묻는다.
~~~
```

### 6) `.claude/settings.json` (하네스 — 권한 기본값)
아래 내용을 그대로 쓴다. 어느 프로젝트에서나 안전한 **읽기 전용 명령**만 미리 허용해 둔다.
(입문자가 `git status` 한 번 돌릴 때마다 허용 프롬프트에 시달리지 않도록 하는 게 목적이다.
`push`·`reset` 같이 되돌리기 어려운 명령은 일부러 넣지 않는다 — 그건 매번 확인받는 게 맞다.)

```json
{
  "permissions": {
    "allow": [
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(ls:*)",
      "PowerShell(git status*)",
      "PowerShell(git diff*)",
      "PowerShell(git log*)",
      "PowerShell(Get-ChildItem*)"
    ]
  }
}
```

## 생성이 끝나면
먼저, 골격 생성을 모두 마친 뒤 마무리 안내 맨 앞에 아래 WOMC 로고 아트를 코드블록(```)으로 감싸 이 모양 그대로 한 번 출력한다.
(일부 파일을 건너뛴 경우에도 아트는 동일하게 맨 앞에 한 번 넣는다.) 아트와 태그라인을 출력하고 한 줄 비운 뒤, 아래 한국어 안내로 이어간다.

```
 __        _____  __  __  ____
 \ \      / / _ \|  \/  |/ ___|
  \ \ /\ / / | | | |\/| | |
   \ V  V /| |_| | |  | | |___
    \_/\_/  \___/|_|  |_|\____|

           works on my claude
```

그다음 한국어로 짧게 다음 단계를 안내한다. 예시:

> ✅ 기본 골격을 만들었어요.
> 다음 순서로 진행하면 됩니다.
> 1) `SPEC.md` 를 채우기 — 어떤 걸 만들지 정한다.
> 2) 기능을 만들 땐 그냥 "○○ 기능 만들자"라고 하세요. `plan-feature` 스킬이 켜져
>    계획(PLAN.md)·진행상태(TASKS.md)를 **그때 자동으로** 만들고, 한 번에 하나씩 구현한 뒤 동작을 같이 확인합니다.
>
> ※ PLAN.md·TASKS.md 는 지금 만들지 않습니다. 필요한 순간에만 생겨 평소 컨텍스트를 가볍게 둡니다.
> ※ MCP가 필요한 프로젝트라면 그때 `.mcp.json` 을 직접 추가하세요.

(SPEC를 대신 채우거나 코드를 짜지 말 것. 여기서 멈춘다.)
