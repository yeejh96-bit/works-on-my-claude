# 기능 추가·수정하기

사용자가 "○○ 기능 만들자" 또는 "○○ 고쳐줘/바꿔줘"라고 하면 이 절차를 따른다. 너(메인)는 **오케스트레이터**다 — 무거운 일은 전문 서브에이전트에 맡기고 결론만 받아 조립한다.

> 아래는 **고정 순서가 아니라 기본 레시피**다. 필요 없는 단계는 건너뛰고, 독립적인 갈래가 있으면 같은 단계를 여러 개 병렬로 돌린다. 위임할지 직접 할지는 그때그때 판단하되, 무거운 일은 위임이 기본값이다.

## 먼저: 작업 크기를 가른다
- **작은 변경** (버그 하나·문구·설정·자명한 리팩터, 파일 1~2개): `PLAN.md`/`TASKS.md` 를 만들지 않는다. (필요하면 `explore` 로 위치만 확인한 뒤) `implement` 로 고치고 → `verify` 로 동작만 확인하고 끝낸다.
- **큰 작업** (새 기능·여러 파일·여러 단계): 아래 0~7 전체 절차를 따른다.
- 애매하면 한 줄로 물어 정한다. 예: "간단히 고칠까요, 아니면 계획을 세워 단계별로 갈까요?"

## 0. 상태 파일 준비 (큰 작업일 때만)
큰 작업이면 `PLAN.md` / `TASKS.md` 가 없을 때 아래 빈 템플릿으로 만든다(내용은 채우지 않는다). 이미 있으면 읽어 현재 상태를 파악한다. (작은 변경이면 이 단계를 건너뛴다.)
이 두 파일은 CLAUDE.md 에 import 하지 않는다 — 이 스킬이 작업 중에만 읽고 갱신한다(평소 컨텍스트를 차지하지 않게).

`PLAN.md` (없을 때만 생성):
~~~markdown
# 구현 계획 (PLAN)

> SPEC을 "무엇을 어떤 순서로 만들지"로 옮긴 큰 그림이다.
> 자잘한 진행 상태는 TASKS.md 가 맡는다. 여기는 순서/마일스톤만.

## 만드는 순서
1. (아직 비어 있음 — SPEC.md 를 채운 뒤 큰 덩어리 순서를 적는다)

## 지금 집중하는 단계
(아직 없음)

## 나중에 / 안 할 것
(아직 없음)
~~~

`TASKS.md` (없을 때만 생성):
~~~markdown
# 작업 목록 (TASKS)

> PLAN의 한 단계를 실제 작업으로 쪼갠 체크리스트다.
> **다른 세션에서 이어 작업해도 되도록**, 각 항목은 그것만 읽고 바로 시작할 수 있게 적는다.
> 표기: `[ ]` 안 함 · `[~]` 하는 중 · `[x]` 끝남

## 지금 하는 일
(아직 없음)

## 할 일 (순서대로)
- [ ] (아직 비어 있음)
  - 손댈 파일: (경로)
  - 이어 쓸 것: (앞 단계가 남긴 파일·함수·설정 이름 — 없으면 "없음")
  - 끝난 것으로 보는 조건: (무엇이 되면 끝인가)
  - 확인 방법: (어떤 명령·동작으로 확인하나 — 모르면 비워두고 검증 단계에서 채운다)

## 끝난 일
(아직 없음)

<!-- 끝난 항목은 이렇게 적는다:
- [x] 항목 이름
  - 남긴 것: 만들어진 파일 경로, 다음 단계가 쓸 함수·설정 이름 (다음 항목의 「이어 쓸 것」에 그대로 옮겨 적는다)
  - 확인 방법: 통과를 확인한 명령
-->
~~~

## 1. SPEC 확인
새 기능이면 `SPEC.md` 에 그 기능이 있는지 보고, 없으면 한 줄로 추가를 제안한다.
버그수정·리팩터면 대개 SPEC 을 안 건드린다 — 동작이나 범위가 바뀌는 변경일 때만 SPEC 갱신이 필요한지 확인한다.

## 2. 조사 (필요하면 — explore 위임)
기능이 어디에 어떻게 얽히는지 모르면 `explore` 서브에이전트에게 조사를 맡긴다.
서로 다른 영역을 봐야 하면 **한 메시지에서 explore 여러 개를 병렬로** 띄운다. 결론(`파일경로:줄번호`)만 돌아와 메인이 가벼워진다.

## 3. 설계 (비자명하면 — plan 위임)
여러 파일이 얽히거나 순서가 애매한 기능이면 `plan` 서브에이전트에게 설계도를 맡긴다(2의 조사 결과를 함께 넘긴다).
돌아온 단계별 계획을 `PLAN.md`/`TASKS.md` 에 반영한다. 각 단계는 파일 1~2개 수준으로 쪼갠다.
`TASKS.md` 에 항목을 적을 때 **「손댈 파일」과 「끝난 것으로 보는 조건」은 그 자리에서 채운다** — 비워두지 않는다. 「이어 쓸 것」은 앞 단계가 끝나야 알 수 있으면 비워두고(4단계에서 채워진다), 「확인 방법」은 5단계에서 채운다. 칸이 비어 있으면 다른 세션에서 그 항목만 읽고 시작할 수 없다.
(간단한 기능이면 이 단계를 건너뛰고 바로 쪼개도 된다.)

## 4. 구현 (implement 위임)
각 단계를 `implement` 서브에이전트에 맡긴다 — 단계 명세 + 손댈 파일 경로 + 따라야 할 패턴(`파일경로:줄번호`)을 함께 넘긴다.
서로 다른 파일을 건드리는 독립 단계는 **여러 개를 병렬로** 맡길 수 있다. 같은 파일을 건드리는 단계는 순서대로.
돌아온 요약으로 `TASKS.md` 를 갱신한다 — 표기를 `[~]` → `[x]` 로 바꾸고, **그 단계가 「남긴 것」(만들어진 파일 경로, 다음 단계가 쓸 함수·설정 이름)을 「끝난 일」에 적은 뒤, 그것을 이어 쓰는 다음 항목의 「이어 쓸 것」 칸에도 옮겨 적는다.** 이 기록은 건너뛰지 않는다 — 다음 세션이 앞 단계의 결과물을 찾는 유일한 단서다. 코드는 implement 가 파일에 직접 쓰고, 메인은 결과를 조립·확인한다.

## 5. 검증 (verify 위임)
구현이 끝나면 확인 방법이 이미 기록돼 있는지 본다(`TASKS.md` 나 `SPEC.md`).
- **기록돼 있으면**: `verify` 서브에이전트에게 위임해 통과/실패만 받는다. 테스트·실행 로그가 이 대화에 쌓이지 않는다.
- **처음이면**: 프로젝트를 살펴 확인 방법을 1~2가지 골라, 쉬운 말로 먼저 제안한다.
  (예: "터미널에서 이 명령 하나를 돌려 '통과'가 뜨는지 보면 됩니다") 사용자는 제안 중 하나를 고르기만 하면 된다.
  확인 방법을 알아내는 일을 사용자에게 떠넘기지 않는다. (실행 명령·테스트는 프로젝트마다 다르므로 미리 박지 않는다.)

확인되면 그 방법을 `TASKS.md` 그 항목의 「확인 방법」 칸에 남긴다 — 다음부터는(다음 세션에서도) 묻지 않고 `verify` 에게 위임한다.

## 6. 검토 (review 위임 — 선택)
한 단계나 기능이 끝나면 `review` 서브에이전트에게 바뀐 부분(diff)을 맡겨 버그·개선점을 받는다.
심각한 것부터 메인이 반영한다(고치는 건 메인 몫 — review 는 지적만 한다).

## 7. 다음으로
한 단계가 동작하는 걸 확인한 뒤에만 다음 단계로 간다.
동작을 확인했으면 사용자에게 커밋을 권한다 — 되돌릴 지점을 남기는 것이다. 사용자는 코딩을 모르므로 "지금까지 한 걸 저장(커밋)해 둘까요?"처럼 쉬운 말로 묻고, 좋다고 하면 메인이 대신 `git add`·`git commit` 한다. `push` 는 자동으로 하지 않는다.
한 기능이 다 끝났을 때 `SPEC.md` 의 「어디서 돌아가나 (배포)」 칸이 비어 있거나 "나중에 정함"이면, 이제 어디에 올릴지 정하자고 쉬운 말로 안내한다 — 배포 방법은 골격에 정해져 있지 않으므로, 그때 프로젝트에 맞게 조사해 1~2가지를 제안한다.
새 세션을 권하기 전에 `TASKS.md` 의 「지금 하는 일」과 「남긴 것」이 최신인지 확인한다 — 다음 세션은 이 파일만 보고 이어간다. 한 기능이 다 끝나면 새 세션으로 시작하길 권한다.
작업 중간에 대화가 무거워지면 사용자에게 `/compact` 를 제안한다.
